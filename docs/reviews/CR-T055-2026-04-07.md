## 变更范围

- `atlasctl tabs capture` 成功后写入 `page_capture` 事件，写入失败时命令显式失败
- `/v1/tabs/context` 成功后写入 `page_capture` 事件，写入失败时返回 500
- `/v1/sidebar/ask` 成功后写入 `qa_turn` 事件，记录 `question`、`answer`、`trace_id` 与引用 URL
- provider 失败路径不写伪 `qa_turn`；memory 写入失败时 ask 主链显式返回错误并写入 runtime error 状态
- 新增 atlasctl 与 daemon 测试，覆盖成功写入、失败不写、memory 写失败显式报错

## 审查结论

- memory 写入只接在当前产品入口层，没有把副作用下沉到 `tabs.Client.Capture` 之类的共享内部路径，变更面保持最小。
- `page_capture` 与 `qa_turn` 都使用可追踪字段落盘：capture 继承 `captured_at`，ask 记录 `trace_id` 与引用 URL。
- 所有失败路径都保持 Debug-First：provider 失败不写假事件，memory 写失败直接中断主链，不存在“响应成功但状态面缺失”的静默降级。
